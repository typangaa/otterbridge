# weir v0.2 — Resilience Wiring & Metrics Collection

> Status: **Planned** · Author: Alex Pang · Target milestone: v0.2.0
> Scope: wire the already-built resilience primitives and metrics counters into
> the live request path, add config surface, persist + expose metrics.

---

## 1. Goal & context

Three resilience primitives (`CircuitBreaker`, `RetryPolicy`/`with_retry`,
`RateLimiter`) and a metrics store (`Metrics` / `BackendMetrics`) are already
**fully implemented and unit-tested** in `src/resilience/` and
`src/observability/`, but **nothing calls them**:

- `Backend::chat()` is invoked directly from `run_chat`, the MCP tool handlers,
  and every engine — no protection, no measurement.
- `WeirServer` already holds a `metrics: Arc<Metrics>` field that is **never
  read or written**.
- `weir status` shows config only; even if metrics were recorded, each CLI
  invocation is a fresh process so counters would always read zero.

v0.2 closes both gaps with a single architectural move (a decorator backend) plus
a metrics persistence layer.

### Non-goals (stay out of scope)

- HTTP transport (v0.3).
- MCP-client backend type / opencode Path C (v0.3).
- Prometheus / OpenTelemetry export (later; JSON snapshot only for now).
- Per-token / cost accounting (the `Usage` struct stays as-is).

## 2. Research findings (verified via weir `agy-gemini`)

### 2.1 Canonical layering order

Confirmed consistent across **Polly (.NET)**, **resilience4j (Java)**, and the
**tower (Rust)** ecosystem:

```
Request → Retry → RateLimiter → CircuitBreaker → Network Call
         (outermost)                            (innermost)
```

Rationale:

1. **Retry is outermost** — each retry attempt re-runs the full admission path
   (rate-limit check + circuit check). If retry were innermost it would bypass
   both and risk a retry storm against a struggling backend.
2. **CircuitBreaker is innermost** — it observes every *physical* attempt, so it
   trips fast and subsequent retries fail-fast instead of hammering the backend.
3. **RateLimiter sits *outside* the breaker** — a local rate-limit rejection must
   **not** count as a backend failure (which would wrongly trip the circuit).
   **Decision: throttle is fail-fast** — a rate-limit rejection returns
   immediately as a dedicated non-retryable error (`WeirError::RateLimited`); it
   does **not** consume the retry budget and does **not** touch the breaker. The
   caller decides what to do. (This deliberately overrides the
   "throttle → backoff" default; see §3.3.)

Reference nesting (matches our existing primitive signatures):

```rust
with_retry(&retry_policy, || async {
    rate_limiter.acquire().await?;                 // admission, per attempt
    circuit_breaker.call(|| async {                // wraps only the call
        inner.chat(req.clone()).await
    }).await
}).await
```

### 2.2 Composition pattern

The decorator pattern (a wrapper struct that itself implements `Backend`) is the
right choice for a single-binary tool — it is fully transparent to every caller
(CLI, MCP tools, all engines) and isolates resilience to one place. We keep the
existing `Arc<dyn Backend>` dynamic dispatch (engines, the server pool, and
`build_backend` all depend on it); the marginal `Box`/`Arc` cost is negligible
next to a network/subprocess round-trip. `tower::Service` was considered and
rejected — its type complexity is not worth it here.

Gotchas to respect (from research + our code):

- Trait methods take `&self`; all mutable resilience state already uses interior
  mutability (`tokio::sync::Mutex`, `AtomicU64`) — safe to share behind `Arc`.
- Record metrics at the **operation level** (end-to-end, outside the retry loop)
  so `avg_latency_ms` reflects user-perceived latency, not per-attempt latency.
- Attach `backend_name` to the wrapper so callers never thread labels through.

## 3. Architecture

### 3.1 The `ResilientBackend` decorator

New file: **`src/resilience/resilient_backend.rs`**

```rust
pub struct ResilientBackend {
    inner: Arc<dyn Backend>,
    name: String,
    retry: RetryPolicy,
    breaker: CircuitBreaker,            // per-backend instance (stateful)
    limiter: Option<RateLimiter>,       // None when rps disabled
    metrics: Arc<BackendMetrics>,       // this backend's counter handle
}

#[async_trait]
impl Backend for ResilientBackend {
    fn name(&self) -> &str { &self.name }

    async fn chat(&self, req: ChatRequest) -> Result<ChatResponse> {
        let start = Instant::now();
        let outcome = with_retry(&self.retry, || async {
            if let Some(l) = &self.limiter { l.acquire().await?; }
            self.breaker.call(|| self.inner.chat(req.clone())).await
        }).await;

        match &outcome {
            Ok(_)  => self.metrics.record_success(start.elapsed().as_millis() as u64),
            Err(_) => self.metrics.record_error(),
        }
        outcome
    }

    async fn health(&self) -> Result<()> { self.inner.health().await }
}
```

Notes:

- `ChatRequest` already derives `Clone` (needed because `with_retry` takes `Fn`,
  not `FnOnce`, and may call the closure multiple times).
- One `CircuitBreaker` and one `RateLimiter` **per backend instance** — state is
  bound to the wrapper, exactly as research recommends.
- Metrics recorded once per logical operation (outside retry), latency only on
  success (matches existing `BackendMetrics::record_success` contract).

### 3.2 Construction — one wrapping point per path

Both construction paths wrap each freshly-built backend before handing it out.

**CLI path** — `build_backend()` in `main.rs` gains the resilience config + a
metrics handle and returns `Arc::new(ResilientBackend::new(inner, ...))`. (See
§5 for the metrics-handle plumbing on the CLI side.)

**Server path** — `WeirServer::new_with_metrics()` wraps each backend as it
populates the pool, pulling each backend's `Arc<BackendMetrics>` from the shared
`self.metrics` store (which is finally used).

Because every engine and tool handler operates on `Arc<dyn Backend>`, **no engine
code changes** — fan-out, pipeline, router, eval-loop, and fusion all inherit
resilience + metrics for free.

### 3.3 Two new fail-fast error variants

Subtlety: `with_retry` currently treats **all** `WeirError::Backend(_)` as
retryable, and both the breaker and the limiter currently return
`WeirError::Backend(...)`. Left as-is, an open circuit would burn the full retry
budget backing off, and a throttle would retry (contrary to the fail-fast
decision). Fix: give each its own variant so retry can distinguish them.

```rust
// error.rs
#[error("circuit open: {0}")]
CircuitOpen(String),
#[error("rate limited: {0}")]
RateLimited(String),
```

- `CircuitBreaker` returns `WeirError::CircuitOpen` (was `Backend`).
- `RateLimiter::acquire` returns `WeirError::RateLimited` (was `Backend`).
- `is_retryable()` in `retry.rs` returns `false` for **both** → fail fast.
- `exit_code_for()` maps both → 2 (system-level).
- Genuine backend failures (HTTP/subprocess/timeout) stay `WeirError::Backend`
  / `WeirError::Http`, which remain retryable — the breaker observes those.

## 4. Config schema changes

Per the decision: **global `[resilience]` block + optional per-backend override.**

### 4.1 New `ResilienceConfig`

`src/config/mod.rs`:

```rust
#[derive(Debug, Clone, Deserialize, Serialize)]
pub struct ResilienceConfig {
    #[serde(default = "default_retry_attempts")]   pub retry_attempts: u32,    // 3
    #[serde(default = "default_base_delay_ms")]    pub base_delay_ms: u64,     // 100
    #[serde(default = "default_max_delay_ms")]     pub max_delay_ms: u64,      // 5000
    #[serde(default = "default_failure_threshold")]pub failure_threshold: u32, // 5
    #[serde(default = "default_recovery_secs")]    pub recovery_secs: u64,     // 30
    #[serde(default = "default_rate_limit_rps")]   pub rate_limit_rps: f64,    // 100.0
}
```

- Top-level `Config` gains `#[serde(default)] pub resilience: ResilienceConfig`.
- `impl Default for ResilienceConfig` mirrors the `default_*` fns so an absent
  `[resilience]` block yields the documented defaults.

### 4.2 Per-backend override

`BackendConfig` gains optional fields (all `Option<T>`, `skip_serializing_if`):

```rust
pub retry_attempts:    Option<u32>,
pub failure_threshold: Option<u32>,
pub recovery_secs:     Option<u64>,
pub rate_limit_rps:    Option<f64>,   // 0.0 disables the limiter for this backend
```

Effective value = per-backend `Some(_)` else global. A resolver helper
`Config::resilience_for(&self, backend_name) -> ResolvedResilience` centralises
the merge so both construction paths agree.

**Rate limiter posture** (per decision): default `rate_limit_rps = 100.0` for
every backend (high safety net); set a backend's `rate_limit_rps = 0` to disable,
or a low value (e.g. `2.0` for `agy`) to throttle a quota-limited cloud backend.
`RateLimiter` is built only when the resolved rps `> 0.0`.

### 4.3 Example weir.toml fragment

```toml
[resilience]
retry_attempts    = 3
base_delay_ms     = 100
max_delay_ms      = 5000
failure_threshold = 5
recovery_secs     = 30
rate_limit_rps    = 100.0   # global safety net

[[backend]]
name           = "agy"
type           = "stdio-cli"
command        = "agy"
args           = ["-p", "{prompt}"]
timeout_secs   = 300
rate_limit_rps = 2.0        # throttle the quota-limited cloud backend

[[backend]]
name           = "hermes-local"
type           = "stdio-cli"
command        = "hermes"
args           = ["-z", "{prompt}"]
rate_limit_rps = 0          # local, never throttle
```

### 4.4 Validation (Layer 1 — syntactic, in `validate.rs`)

Add a `validate_resilience()` pass:

- `retry_attempts >= 1` (0 would make `with_retry` panic on the `expect`).
- `max_delay_ms >= base_delay_ms`.
- `failure_threshold >= 1`.
- `rate_limit_rps >= 0.0` (0 = disabled; negative rejected).
- Same checks on each per-backend override that is `Some`.

## 5. Metrics: collection, persistence, exposure

Per the decision: **MCP tool + persist to disk; `weir status` reads disk.**

### 5.1 Collection

Recording lives entirely inside `ResilientBackend::chat()` (§3.1) via the
`Arc<BackendMetrics>` handle. The server's existing `Arc<Metrics>` store becomes
the source of those handles (`Metrics::get_or_create(name)` at wrap time).

CLI plumbing: `build_backend()` takes an `&Arc<Metrics>`. `run_chat` /
`run_workflow` create a process-local `Arc<Metrics>` at the top, pass it down,
and flush it to disk on exit (so even oneshot CLI calls contribute samples).

### 5.2 Persistence

New file: **`src/observability/persist.rs`**

- Path: `~/.local/state/weir/metrics.json` (XDG state dir; fall back to
  `~/.weir/metrics.json` if `$HOME` resolution is all that's available).
- Format: the existing `Metrics::snapshot()` JSON, plus a `"version"` and
  `"updated_unix"` field.
- **Merge-on-write** (load-add-store): read existing file, add the current
  process's counters to the stored totals, write back. This makes oneshot CLI
  calls accumulate across invocations and lets the long-running `serve` process
  contribute too. Write atomically (temp file + rename) to avoid torn reads.
  - **Counters merge additively** (req/err/latency-sum/latency-count summed).
  - **Circuit state is last-writer-wins and written ONLY by the `serve`
    process** — a fresh CLI breaker is always `Closed` and would clobber real
    state, so CLI flushes counters but leaves the `circuit_state` block
    untouched. See §5.5.
- The `serve` process flushes periodically (e.g. a `tokio::spawn` ticking every
  30 s) **and** on graceful shutdown; CLI flushes once before exit.

> Concurrency note: a `serve` process and a CLI call can race on the file. The
> temp-file + atomic-rename keeps each write internally consistent; a lost update
> under simultaneous writes is acceptable for advisory metrics. (A file-lock can
> be added in v0.4 if it matters.)

### 5.3 `weir status` reads disk

`Command::Status` additionally loads `metrics.json` (if present) and prints a
per-backend table, including last-known circuit state (see §5.5):

```
Backends:  5 configured
  - agy           142 req,  3 err, avg 1840.0 ms   [CLOSED]
  - agy-gemini     20 req, 18 err, avg 2110.0 ms   [OPEN, recovered?]
  - hermes-local   89 req,  0 err, avg  410.2 ms   [CLOSED]
  ...
(circuit state as of 14:31:40 UTC, written by serve process)
```

`--json` status embeds the snapshot under a `"metrics"` key. Missing file → omit
the metrics section (no error).

### 5.4 New `metrics` MCP tool

Add to `src/server/tools.rs`:

```rust
#[tool(description = "Return a JSON snapshot of per-backend request/error/latency metrics.")]
pub async fn metrics(&self) -> ToolResult {
    serde_json::to_string(&self.metrics.snapshot().await)
        .map_err(|e| rmcp::ErrorData::internal_error(e.to_string(), None))
}
```

Reflects live server-lifetime counters (independent of the disk file). The live
snapshot also carries each backend's **current** circuit state (see §5.5).

### 5.5 Circuit-breaker state exposure

Decision: expose breaker state (`closed` / `open` / `half-open`) alongside the
counters. Implementation:

- `CircuitBreaker` gains `pub async fn state(&self) -> CircuitState`, where
  `CircuitState` is a small public enum `{ Closed, Open { until: Instant },
  HalfOpen }` (mirrors the private `State`).
- **Live path (accurate):** the `metrics` MCP tool and the `serve` process iterate
  the `ResilientBackend` pool, query each breaker's `state()`, and emit it next to
  the counters. This is always truthful because the breakers are in-memory.
- **Disk path (best-effort):** on each `serve` flush, the current state + the
  `until` timestamp are written to a `circuit_state` block in `metrics.json`.
  `weir status` (a separate process) reads it and renders it honestly:
  - `Closed` → `[CLOSED]`
  - `Open { until }` where `until` is in the future → `[OPEN until HH:MM]`
  - `Open { until }` where `until` has elapsed → `[OPEN, recovered?]` (the next
    real request will probe HalfOpen; the disk file just hasn't been updated yet)
  - state block older than, say, 2× the flush interval → suffix `(stale)`
- CLI oneshot processes never write the `circuit_state` block (§5.2) — their
  fresh `Closed` breakers carry no signal.

Rationale for the asymmetry: counters are durable cumulative facts (safe to merge
across any process); circuit state is a *live* property of a running gateway, so
the disk copy is explicitly framed as a possibly-stale snapshot from `serve`.

## 6. Related gap surfaced during planning — stdio-cli timeout

`StdioCliBackend` **does not enforce `timeout_secs`** (only `OpenaiCompatBackend`
does). A hung subprocess would hang forever, defeating the circuit breaker (which
only trips on *returned* errors). Fold a minimal fix into v0.2:

- Wrap the `child.wait_with_output()` await in `tokio::time::timeout(
  Duration::from_secs(cfg.timeout_secs), …)`; on elapse, kill the child and
  return `WeirError::Backend("backend '{name}' timed out after {n}s")` (retryable
  → the breaker sees a real failure). Keep `.stdin(Stdio::null())` (hard
  constraint #2).

## 8. Testing plan

**Unit (target +10–14 tests, ~32 → ~45):**

- `resilient_backend`: success records latency; error records error; retryable
  error retries then succeeds; circuit opens after N failures → fail-fast (no
  retry); **rate-limit rejection is fail-fast (NOT retried)** — assert the inner
  backend is called exactly once and the error is `RateLimited`; rps=0 → no
  limiter built (no throttling); breaker `state()` reports Open after tripping.
- `config`: per-backend override beats global; absent `[resilience]` → defaults;
  `resilience_for()` merge correctness.
- `validate`: each new rule rejects its bad input (retry_attempts=0,
  max<base, negative rps).
- `persist`: write→read round-trip; merge adds onto existing totals; missing file
  → empty snapshot; atomic rename leaves no temp file.
- `stdio_cli`: a `sleep 5` command with `timeout_secs=1` returns a timeout error
  (gated behind a cfg so CI without the binary still passes, or use `sleep`).

**Smoke (manual, after `cargo build --release`):**

```sh
cargo test && cargo clippy -- -D warnings
weir validate --config weir.example.toml --json          # resilience block valid
weir --config ~/.config/weir/weir.toml chat agy "ping"   # records a sample
weir --config ~/.config/weir/weir.toml status            # shows metrics table
cat ~/.local/state/weir/metrics.json                     # persisted snapshot
# MCP metrics tool:
printf '%s\n' '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"metrics","arguments":{}}}' \
  | weir serve --config ~/.config/weir/weir.toml   # (after initialize handshake)
```

## 10. Open questions / risks

- **Metrics file race** (serve + CLI concurrent writes): mitigated by atomic
  rename; lost-update accepted for advisory metrics. Revisit with a lock in v0.4
  if it proves noisy.
- **Rate limiter on local backends**: default 100 rps is effectively unlimited
  for interactive use; documented how to set `rate_limit_rps = 0` to fully
  disable. Confirm 100 is high enough for fan-out bursts (4-way fan-out = 4
  concurrent, well under 100).
- **Fail-fast throttle UX**: a throttled request returns `RateLimited`
  immediately with no retry (per decision). For a quota-limited cloud backend set
  to a low rps (e.g. `agy` at 2.0), a burst beyond the bucket surfaces an error to
  the caller rather than silently waiting. This is intentional (predictable,
  no hidden latency); callers/workflows that prefer smoothing can lower
  concurrency instead. Revisit a `blocking-wait` mode in a later version if the
  hard failure proves annoying in practice.
- **CLI oneshot resilience**: circuit breaker / rate limiter state does not
  persist across CLI invocations (each is a fresh process) — only retry is
  meaningful there. This is expected and acceptable; the full stack matters for
  the long-running `serve` process. Documented, not a bug.
- **`async-trait` boxing**: unchanged from today; decorator adds one extra boxed
  future per call — negligible.
